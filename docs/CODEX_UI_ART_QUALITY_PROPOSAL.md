# 五行迷宮塔防 — Codex 美術與 UI 品質修正案

> 文件狀態：核心範圍已實作；其餘仍為選修提案
> 建立日期：2026-07-12  
> 核准／實作日期：2026-07-19（共用視覺基礎、戰鬥 HUD、主選單）
> 適用範圍：美術方向、UI 視覺、資訊層級、字體與操作回饋  
> 明確不包含：玩法與數值改動、平台功能；關卡、卡牌、彈窗、結算與羅盤圖鑑須另行核准

## 1. 修正目標

本次改版的核心不是單純「把畫面調亮」，而是讓遊戲從深色 Web Prototype，轉為具有完整世界觀、遠距離也容易閱讀的原生遊戲介面；保留 Console Game 的視覺完成度，但不納入手把操作。

目標體驗：

- 第一眼是「明亮的東方奇幻塔防」，不是深色網頁 Dashboard。
- 戰場、塔、怪物是視覺主角，UI 退居第二層但仍清楚易讀。
- 按鈕具有實體材質、邊框、壓下與聚焦狀態，不依賴網頁式 Hover。
- 玩家不必閱讀長段文字，也能藉由圖示、數值、顏色和短標籤做決策。
- 以電視或掌機距離觀看時，重要資訊仍能辨識。

## 2. 現況診斷

目前 UI 已有像素字型、像素描邊與部分 9-slice 概念，但整體仍呈現 Web Prototype 感，原因主要如下：

1. 大量使用深藍黑底、半透明面板、背景模糊、CSS 圓角及柔和陰影，語彙接近 SaaS Dashboard。
2. 主選單、關卡卡片、HUD、Codex 與彈窗使用相似的矩形卡片結構，缺少場景身分。
3. 漸層文字、細框線與 Hover 上浮是網頁常見反饋，缺少原生遊戲按鍵的重量與狀態感。
4. 多處字級落在 `0.62rem–0.85rem`，塔名、價格、說明與前置條件尤其偏小。
5. 教學及天賦卡同時展示名稱、說明、條件、費用與提示，資訊密度過高。
6. Emoji 與像素圖示混用，圖示的輪廓、尺寸和視覺重量不一致。
7. 戰場背景偏暗，元素光效也使用高飽和螢光，長時間觀看容易疲勞，且削弱塔與怪物的剪影辨識。

## 3. 建議美術方向

### 3.1 定位：明亮東方奇幻 × 原生遊戲像素 UI

參考 Kingdom Rush 的不是造型本身，而是以下方法：

- 戰場保持明亮、溫暖、輪廓清楚。
- 操作按鈕大、圖示優先、狀態明確。
- 資訊面板使用具有世界觀的實體材質，而非漂浮的玻璃卡片。
- 戰鬥畫面只常駐真正必要的資訊，其餘在選取或暫停時出現。

本作的差異化應保留在五行配色、東方輪廓與經脈意象。材質不追求寫實木材、青銅或玉石質感，而以低風格化像素塊、清楚剪影、明暗分層與一致描邊建立辨識度。禁止直接複製 Kingdom Rush 的中世紀羊皮紙、盾牌或按鈕輪廓。

### 3.2 情緒關鍵字

`明亮`、`溫暖`、`輪廓清楚`、`原生像素`、`五行能量`、`山水奇幻`、`戰場清楚`、`按鍵有重量`

### 3.3 建議色彩架構

色彩分為「世界色」、「介面材質色」、「五行功能色」三層，避免所有元素都發光競爭注意力。

| 用途 | 建議色 | 使用原則 |
|---|---|---|
| 戰場主底 | `#D8C98E`、`#A9C978`、`#8CB8B0` | 依地圖主題選一組中高明度自然色 |
| 暖色像素面板 | `#6E442B` / `#A96F3C` / `#E2B96F` | 三階色塊形成厚度，不描繪寫實材質 |
| 冷色像素面板 | `#2E6F68` / `#55A89A` / `#BFE3C0` | 用於天賦、法術、五行資訊 |
| 明亮資訊內頁 | `#F2DFA7` / `#5A3826` | Codex、Tip、說明與結算頁 |
| 主要文字 | `#FFF3C4` 或 `#3A251B` | 依深／淺底切換，確保高對比 |
| 次要文字 | `#D7C99E` 或 `#69513D` | 不以低透明度代替次要色 |
| 聚焦狀態 | `#FFE26A` + `#FFF3B0` | 鍵盤焦點框、目前選項、重要 CTA |
| 危險狀態 | `#D9473F` + `#FFD0A8` | 只用於生命危急、刪除、退出 |

五行色維持金、木、水、火、土的辨識，但整體提高明度、降低霓虹感：

- 金：暖金 `#F2C14E`
- 木：嫩葉綠 `#69B84A`
- 水：湖水藍 `#4CA7D8`
- 火：朱橙 `#EE6638`
- 土：陶土棕 `#B77A45`
- 陰：靛紫 `#6656A8`
- 陽：日光黃 `#F5D85A`

## 4. 原生遊戲 UI 設計語言

### 4.1 移除 Web 愼用項目

以下語彙應逐步淘汰：

- `backdrop-filter` 玻璃模糊。
- 12–16px 通用圓角卡片。
- 漸層裁切文字作為標題。
- 細 1px 灰色邊框。
- Hover 上浮、柔和長陰影作為主要反饋。
- 純文字超連結式按鈕。
- 瀏覽器原生 `confirm`、表單輸入框及捲軸外觀。

### 4.2 統一元件語彙

所有可操作元件改採共通狀態：

| 狀態 | 視覺表現 |
|---|---|
| Normal | 厚 2–3px 深色外框、材質底、上緣亮邊 |
| Focused | 金色像素焦點框、左右指標或角標、輕微呼吸動畫 |
| Pressed | 整體向下 2px、上緣亮邊消失、下緣陰影縮短 |
| Disabled | 降低飽和度，不只降低透明度；顯示鎖或原因 Tip |
| Selected | 元素色底盤、清楚勾選／插旗標記，與 Focused 可同時存在 |

按鈕改為 9-slice 像素框或 CSS 像素階梯框。主要按鈕至少 56px 高，次要按鈕至少 48px；圖示與文字必須有固定位置，不因字數改變而跳動。

### 4.3 PC／Mobile 雙輸入版面行為

- 不規劃手把支援；第一階段以 PC 滑鼠／鍵盤與 Mobile 觸控為目標。
- PC 可保留 Hover 與快捷鍵，但 Hover 只能提供補充資訊，不能是唯一操作提示。
- Mobile 所有必要資訊皆可透過點按、長按或明確的 Tip 按鈕取得，不依賴滑鼠懸停。
- 共用視覺元件，依輸入模式切換尺寸、排列與提示；避免維護兩套完全獨立 UI。
- Mobile 主要觸控區至少 48×48 CSS px，重要操作建議 56×56 CSS px。
- 戰鬥操作保留既有單指平移、點按建塔與雙指縮放能力，改版不可破壞座標換算。
- Modal 開啟後阻擋背景誤觸，關閉後保留原本選取狀態。
- 不使用狹窄橫向捲軸尋找內容；PC 可用分頁／快捷鍵，Mobile 可用大型頁籤／滑動切頁。

## 5. 字體與資訊減量規範

### 5.1 字級下限

以目前 1024px 級遊戲容器為基準：

| 層級 | 建議尺寸 | 用途 |
|---|---:|---|
| Display | 36–48px | 主選單標題、勝敗標題 |
| H1 | 28–32px | 畫面名稱 |
| H2 | 22–26px | 面板／章節名稱 |
| Body | 18–20px | 一般可讀文字 |
| Label | 16–18px | 塔名、費用、按鍵提示 |
| Minimum | 15–16px | 次要狀態；禁止再小 |

字體以 PC 與 Mobile 的正確辨識為最高原則，不強制使用像素字。主字型優先採輪廓清楚、中文字重完整且跨平台穩定的黑體；若測試證明像素字在標題或數字上同樣清楚，可局部作為裝飾字體，否則全介面統一使用清楚黑體。字體的遊戲感主要由粗細、描邊、色塊與版面建立，不以犧牲可讀性換取像素感。

### 5.2 資訊三層化

- 常駐：生命、金幣、波次、目前工具、下一波／速度。
- 選取時：塔的傷害、射程、攻速、元素、升級與出售。
- Tip／詳情：完整技能敘述、克制公式、前置條件、背景敘事。

### 5.3 文案刪減原則

- 按鈕使用動詞或名詞，最多 2–4 個中文字，例如「開戰」「合成」「圖鑑」。
- 同一資訊不以圖示、標題與描述重複三次。
- 數值用圖示＋數字呈現；只有數值含義不明時加短標籤。
- 教學每一步只要求一個動作，主句不超過兩行。
- Tip 延遲約 300–500ms 出現，最多 3 行；進階內容用「詳情」開啟。

## 6. 各畫面修正方案

### 6.1 主選單

- 改為全畫面場景插畫／動態像素背景，選單嵌入世界而非置中網頁按鈕列。
- 標題使用實色像素描邊，不使用彩虹漸層裁字。
- 主操作保留「開始」「天賦」「設定／離開」三層；低頻項目收入次選單。
- 目前天賦點以小型徽章呈現，不用一行說明文字。
- 「開始」維持唯一最高權重操作；PC 顯示鍵盤選取框，Mobile 顯示足夠大的觸控區。

### 6.2 關卡選擇

- 網頁卡片 Grid 改為「地圖卷軸／節點地圖＋右側關卡資訊板」。
- 常態只顯示關卡名稱、難度徽章與完成狀態。
- 起終點座標、障礙物數量等開發資訊移出正式玩家介面；測試模式才顯示。
- 自訂地圖的刪除改為詳情選單內的危險操作，避免卡片上常駐小按鈕。

### 6.3 戰鬥 HUD

- 上方 HUD 改為三個實體徽章：基地生命、金幣、波次；以大數字為主。
- 塔選擇列改為大圖示輪盤或橫向快捷列，卡面只保留圖示、名稱、費用。
- 選中塔後才展開屬性面板；未選取時讓出戰場空間。
- 「下一波」成為唯一高權重 CTA；合成、圖鑑、速度使用功能圖示與按鍵提示。
- 五行羅盤可維持，但縮成狀態元件；詳細克制說明移至 Tip／Codex。
- 移除 Canvas 外框的網頁容器感，使戰場延伸到畫面邊緣或嵌入明確的機關台框架。

### 6.4 天賦頁

- 採「太極核心＋人體五行經脈＋穴位節點」而非橫向網頁卡片列，呼應既有 Roadmap 的敘事方向。
- 節點常態只顯示圖示、名稱、等級；描述與花費放右側詳情板。
- 可解鎖、已解鎖、未達條件使用形狀＋顏色雙重區別。
- 分支切換改為大型頁籤；PC 可用快捷鍵切換，Mobile 直接點按或滑動切頁，不使用小型 Nav Button。
- 背景以明亮米紙、淡青人體經脈剪影和五行器官色塊構成，避免醫療寫實感。
- 首次進入只顯示太極核心與第一圈基礎穴位；隨進度逐步點亮五臟／五行分支，降低一次理解整張圖的負擔。
- 人體輪廓只作空間導引，不要求玩家理解真實中醫經絡；每條分支仍需以圖示、名稱與短句說明其玩法功能。

### 6.5 Roguelike 卡牌、Codex 與彈窗

- 卡牌保留稀有度，但避免大量發光；以框材質、角標與一處主色區分。
- 卡牌正面只顯示名稱、主圖示、單句效果與關鍵數字。
- 完整機制、條件與已學習詞條放 Tip 或詳情頁。
- Codex 改為紙本／竹簡式分頁，使用固定頁籤與圖示索引。
- 確認彈窗改為遊戲內 Modal，預設焦點放在安全選項；不再使用瀏覽器原生確認框。

### 6.6 結算畫面

- 先呈現勝敗、到達波次、獲得天賦點三項核心結果。
- DPS、最高連殺等資料放入「戰鬥詳情」折疊頁。
- 下一步按鈕最多兩個，主次層級明確。
- 勝利與失敗使用不同場景插畫、音效和色溫，而非只替換標題顏色。

## 7. 圖示與像素資產規格

- 正式 UI 不混用系統 Emoji；逐步換成同一套 16×16、24×24 或 32×32 像素 Icon。
- 同一尺寸採相同描邊粗細、光源方向與陰影色。
- 圖示輪廓必須在灰階下仍可辨識，元素色只做第二辨識層。
- 需要外置圖片時維持既有防禦性載入；失敗時回退到原生像素矩陣或文字圖示，不中斷操作。
- 9-slice 面板建議建立木、玉、紙、危險四套材質，而非每個畫面獨立設計。

## 8. 建議執行階段

### Phase A — 視覺基準與 UI Token

1. 先製作主選單、標準示範關卡的戰鬥 HUD、天賦頁三張靜態 Mockup。
2. 確認主色調、材質方向、字型組合與按鈕輪廓。
3. 建立統一色票、字級、間距、描邊、陰影和焦點 Token。

驗收：三個畫面看起來屬於同一款遊戲；縮到掌機尺寸仍能讀取核心資訊。

### Phase B — 共用遊戲 UI 元件

1. 建立 Button、Icon Button、Tab、Badge、Tooltip、Modal、Panel、Card 等共用元件。
2. 完成 Normal／Hovered／Focused／Pressed／Disabled／Selected 狀態。
3. 加入鍵盤焦點、滑鼠操作與 Mobile 觸控狀態。

驗收：PC 不用 Hover 也能辨認可否操作；Mobile 點按後能清楚辨認選取與按下結果。

### Phase C — 戰鬥 HUD 優先改造

1. 重排常駐資訊。
2. 放大塔快捷列與關鍵操作。
3. 將詳細數據移到選取面板與 Tip。
4. 明亮化地圖 Tile，降低不必要 Glow。

驗收：玩家可在 3 秒內找到生命、金幣、波次、下一波與目前選塔。

### Phase D — 選單、關卡、天賦與 Codex

依主選單 → 關卡選擇 → 天賦 → 卡牌／Codex → 結算順序替換舊介面，避免一次全面重寫。

### Phase E — 視覺一致性與效能

1. 移除殘留 Emoji、玻璃面板、圓角卡片與過小字級。
2. 檢查四套地圖主題下的文字與單位辨識度。
3. 壓測 50 塔／50 怪／大量粒子場景，確保 UI 改版不增加 Canvas 瓶頸。
4. 進行鍵盤、滑鼠與觸控回歸測試。

## 9. 驗收量化標準

- 正式玩家介面最小字級不得低於 15px，主要操作標籤不得低於 16px。
- 主要按鈕高度至少 56px，次要按鈕至少 48px。
- 所有互動元件具備非 Hover 的 Focused／Selected 與 Pressed 狀態。
- 戰鬥 HUD 常駐文字項目減少至少 35%。
- 教學單步主文不超過兩行，額外知識移至 Tip。
- UI 色彩對比至少以 WCAG AA 作輔助檢查，但像素輪廓和實際遊戲距離測試為最終準則。
- 在 1280×720、1920×1080 與掌機尺寸三種畫面下，不依靠瀏覽器縮放即可讀取核心資訊。
- 不出現瀏覽器原生確認框、未造型表單、系統 Emoji 或 Web 捲軸。
- 60 FPS 壓測不得因新增 UI 裝飾產生明顯退化。

## 10. 決策狀態與實作前閘門

### 10.1 已確認

1. **主材質**：採低風格化像素表現，以輪廓、三階明暗與一致描邊為主，不追求寫實材質。
2. **輸入平台**：不支援手把；規劃 PC 與 Mobile 兩種版面／操作狀態，保留既有觸控操作。
3. **天賦頁**：採太極核心＋人體五行經脈；以逐步開放與清楚功能標籤避免難以理解。
4. **字體策略**：跨裝置辨識優先，不強制使用像素字；優先選用清楚黑體，像素字僅在實機測試清楚時局部使用。
5. **戰場亮度**：大幅轉為日間暖色；夜色、星空與雷雨保留為特殊關卡，不作為全遊戲基準色。
6. **Mobile 方向**：第一階段只支援橫向，不建立直向版戰鬥構圖。
7. **主選單主視覺**：採「通天塔剪影＋明亮山水遠景」。
8. **地圖改造順序**：先完成一個標準示範關卡，通過視覺與操作驗收後，再逐套適配其他地圖主題。

### 10.2 尚待 Mockup 驗證（非方向阻擋）

以下不需要在概念稿前另行決策，可在第一輪 Mockup 比較後定案：

1. 正式黑體字型及字重組合，以 Windows、Android、iOS 的實際顯示與載入 fallback 為準。
2. 標準示範關卡的確切暖色色票與戰場／HUD 對比強度。
3. PC 與 Mobile 橫向版的 UI 切換寬度，以及窄螢幕上的塔快捷列收合方式。
4. 通天塔剪影在主選單中的比例、角色／怪物點綴量與背景動態幅度。
5. 天賦頁第一圈顯示的節點數量與逐步開放節奏。

目前已無阻擋第一輪 Mockup 的方向問題。下一階段仍應只製作視覺稿，待確認後才進入 CSS、DOM 與 Canvas 改造。

## 11. 參考來源與取用原則
