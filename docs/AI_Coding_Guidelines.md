# AI 協同開發指南 (AI Coding Guidelines)

本指南旨在為參與本專案開發的 AI 助手（包括本助手與後續的 AI 代理）提供統一的代碼風格、架構規範與開發流程指引，以確保在 **Vibecoding** 快速迭代模式下，專案代碼仍能保持高可維護性與清晰結構。

---

## 核心開發原則

### 1. 簡潔至上 (KISS Principle)
*   **原則**：拒絕過度設計。不要為尚未實現的系統預留過多接口，亦不要編寫複雜的防禦性代碼。
*   **做法**：
    *   一個函數只做一件事（Single Responsibility）。
    *   避免深層巢狀（Nested）邏輯，多使用「衛句（Guard Clauses）」提早返回。
    *   優先使用簡單的資料結構（如 Array, Dictionary/Map），而非過度複雜的自訂類別。

### 2. 資料與邏輯分離 (Data-Driven Design)
*   **原則**：遊戲數值（如防禦塔傷害、升級花費、怪物血量、移動速度）不應寫死（Hardcode）在邏輯代碼中。
*   **做法**：
    *   在專案中建立 `data/` 或 `config/` 資料夾，使用 JSON 或 CSV 存放數值。
    *   遊戲代碼通過載入這些設定檔來初始化對象。

### 3. 模組化與狀態機 (State Machine)
*   **原則**：對於行為複雜的對象（如怪物、防禦塔、遊戲流程管理），應使用簡單的狀態機（Finite State Machine）來管理其行為。
*   **怪物狀態範例**：`SPAWNING` (生成中) -> `MOVING_TO_WAYPOINT` (前往檢查點) -> `WAITING_AT_WAYPOINT` (在檢查點停留) -> `DYING` (死亡/消滅) -> `REACHED_BASE` (抵達基地)。
*   **防禦塔狀態範例**：`IDLE` (待機/尋找目標) -> `ATTACKING` (攻擊中/冷卻中)。

## 技術規範 (以多檢查點迷宮塔防為例)

### 1. 尋路優化 (Pathfinding Optimization)
*   **問題**：若地圖上有 100 隻怪物，且玩家放置了一個防禦塔，如果所有怪物在同一幀重新計算 A* 尋路，會造成嚴重的畫面卡頓。
*   **規範**：
    *   **事件驅動尋路**：只有當玩家「成功放置或拆除塔」時，才觸發全局尋路地圖更新。
    *   **分幀重新尋路 (Staggered Pathfinding)**：當需要更新所有怪物的路徑時，限制每幀最多更新 N 隻怪物的尋路，或者讓怪物在下一次需要移動到新網格時才更新路徑。
    *   **局部尋路更新**：如果怪物已經過了檢查點 1，且玩家只改變了起點到點 1 的迷宮，那麼這隻怪物不需要重新尋路。

### 2. 座標系統統一
*   **規範**：
    *   **網格座標 (Grid Coordinates)**：如 `(x, y)` 均為整數，代表迷宮中的格子位置（用於尋路與塔的擺放）。
    *   **世界座標 (World/Pixel Coordinates)**：如 `(px, py)` 實數，代表畫面上實際的像素位置（用於渲染與物理碰撞）。
    *   所有涉及尋路與碰撞的計算，應在網格座標系進行；只有在渲染和移動插值時，才轉換為世界座標。

## AI 協同工作流 (AI Workflow in Vibecoding)

當您（AI）接收到代碼編寫任務時，請遵循以下步驟：
1.  **閱讀現有文檔**：先讀取 `docs/` 下的 GDD 與技術規格，確保理解遊戲規則。
2.  **小步快跑 (Incremental Updates)**：每次修改不宜過大。先實現核心邏輯，運行測試後，再添加視覺效果或次要功能。
3.  **註解與文檔同步**：在修改核心邏輯後，及時更新 `docs/` 下的相關規格文檔，並將更新摘要寫入專案根目錄的 `Update.txt` 中。
